## DOCUMENTATION

### **STRICTLY PROHIBITED**
- Business value assessments, ROI calculations, or monetary estimates
- Quantitative business metrics, KPIs, or financial projections
- Market analysis, competitive positioning, or business strategy content
- Cost-benefit analysis or economic impact statements
- Time estimates, story points, sprint planning, or delivery timelines

### General
- **Replace all {IDENTIFIER} placeholders** with actual values and hierarchy
- **Include working links** to parent and child documents with proper URL encoding
- **Use technical descriptions** focusing exclusively on implementation impact and functionality
- **Include navigation headers** pointing to related documentation
- **Create linked indexes** with proper markdown links to all related documents
- **Apply appropriate framework** based on documentation level
- **Focus on technical scope** only (architectural, implementation, engineering)
- **Limit key items** to maintain focus and readability
- **Include clear dependencies** with specific IDs and links where applicable
- **Track status progression** appropriate to documentation level
- **Document checklists** with appropriate state transitions
- **Avoid business language** keeping all descriptions purely technical

### Maintenance
- **Validate cross-references** ensuring all documents have functional bidirectional links
- **Maintain link integrity** with URL encoding and path accuracy for all cross-references
- **Track progress status** keeping status fields current with actual implementation state
- **Follow status progression** rules appropriate for documentation level
- **Use version control** to maintain documentation alongside code
- **Update immediately** when status changes occur
